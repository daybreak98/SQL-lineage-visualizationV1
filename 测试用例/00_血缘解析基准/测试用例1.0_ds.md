# SQL 血缘解析测试用例 v1.0（结构化数据版）

## 0. 文档目的

本文档为 SQL 血缘解析工作台提供**可程式化断言**的 Golden Case 测试集。每个用例包含：

- **完整 SQL**（可直接执行于 SQLGlot 解析器）
- **元数据上下文**（该用例依赖的表和字段定义）
- **结构化血缘结果**：以 `output_columns[].processing_chain[]` 为核心，**逐字段**追溯从源表列→表达式→别名→CTE/子查询传递→最终输出的完整加工链路
- **图谱节点/边**：可直接比对 GraphViewModel 快照
- **预期诊断**：按 DiagnosticCode 枚举断言

所有 JSON 结构均为 `snake_case`（后端 Python 侧格式），前端侧由序列化层自动转换为 `camelCase`。

## 2. 血缘输出格式说明

### 2.1 `processing_chain` 结构

每个输出字段的加工链路是一个有序数组，`step` 从 1 递增，表达数据从源表到最终输出的完整路径：

```json
{
  "step": 1,
  "scope": "source_table | main_query | cte_<name> | subquery_<alias>",
  "node_type": "source_column | expression | aggregate_expression | alias | output_column | literal | window_expression | case_condition | case_result | join_key | filter_predicate | group_by_key",
  "node_id": "唯一标识",
  "node_label": "人类可读标签",
  "transform": "direct_projection | alias | aggregation_<func> | expression_<op> | case_when | window_<func> | join_condition | filter_condition | group_by_key",
  "metadata": { "data_type": "可选字段类型", "comment": "可选字段注释" }
}
```

### 2.2 图谱节点/边

`graph_nodes` 和 `graph_edges` 为扁平的图谱模型，`node_id` 与 `processing_chain` 中的 `node_id` 一致，用于前端画布渲染和快照测试断言。

### 2.3 诊断格式

```json
{
  "code": "DiagnosticCode枚举值",
  "level": "error | warning | info",
  "message": "人类可读消息",
  "suggestion": "修复建议"
}
```

### TC05｜ambiguous_column（字段歧义）

**目标**：join 场景下，多表共有的无前缀字段应标记为 `AMBIGUOUS_COLUMN`，仅单表存在的字段可正常消歧。

**SQL：**
```sql

FROM default.orders a
JOIN default.users b ON a.user_id = b.user_id
```

> **歧义说明**：`user_id` 同时存在于 `orders`（别名为 `a`）和 `users`（别名为 `b`），无法自动消歧。`order_amt` 仅存在于 `orders`，可唯一确定来源为 `a.order_amt`。

### TC13｜complex_nested（CTE 嵌套 + 子查询 + JOIN + 聚合 + 窗口 + CASE WHEN + HAVING + ORDER BY）

**目标**：这是**复合场景压力测试**，验证系统能否在一条 SQL 中同时正确处理：

- 两层 CTE（`user_stats` → `ranked_users`）
- CTE 内的聚合（COUNT / SUM）和 GROUP BY
- CTE 间的 JOIN（user_stats + users）
- 窗口函数（ROW_NUMBER）在第二层 CTE 中
- CASE WHEN 在主查询中消费窗口函数结果
- WHERE 在主查询中过滤 CTE 输出
- ORDER BY 窗口函数结果

**SQL：**
```sql
WITH user_stats AS (
    SELECT
        user_id,
        COUNT(*)        AS order_cnt,
        SUM(order_amt)  AS total_amt
    FROM default.orders
    WHERE dt BETWEEN '2026-05-01' AND '2026-05-28'
    GROUP BY user_id
),
ranked_users AS (
    SELECT
        us.user_id,
        us.order_cnt,
        us.total_amt,
        u.user_name,
        u.user_level,
        ROW_NUMBER() OVER (ORDER BY us.total_amt DESC) AS rank
    FROM user_stats us
    JOIN default.users u ON us.user_id = u.user_id
)
SELECT
    user_id,
    user_name,
    user_level,
    order_cnt,
    total_amt,
    rank,
    CASE
        WHEN rank <= 10  THEN 'top10'
        WHEN rank <= 50  THEN 'top50'
        ELSE 'normal'
    END AS user_tier
FROM ranked_users
WHERE order_cnt >= 5
ORDER BY rank
```

## 5. 断言策略

测试代码断言分为四个层级：

| 层级 | 断言内容 | 示例 |
|---|---|---|
| L1 结构断言 | `output_columns` 数量、名称正确 | `assert len(result.output_columns) == 3` |
| L2 链路断言 | 每个字段的 `processing_chain` step 数量、每步的 `node_type` 和 `transform` | `assert chain[2].transform == "aggregation_sum"` |
| L3 节点/边断言 | `graph_nodes` 和 `graph_edges` 的结构与快照一致 | `assert snapshot == expected_nodes` |
| L4 诊断断言 | `expected_diagnostics` 的 `code`、`level` 完全匹配 | `assert diagnostics[0].code == "AMBIGUOUS_COLUMN"` |

> **注意**：`node_id` 不要求精确匹配（只需保证唯一性和可追踪性），但 `node_type`、`transform`、`scope` 必须精确匹配。
