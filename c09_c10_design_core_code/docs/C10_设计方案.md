# C10｜Golden Case 回归冻结设计方案

## 1. 目标

C10 不是新增复杂能力，而是冻结 C00-C09 已完成能力，防止后续开发把核心能力改坏。

C10 要保证：

```text
1. 后端 analyze API 稳定。
2. graph_view_model.nodes / edges 不出现悬空边。
3. C08 的 output_column -> Query Result 链路稳定。
4. C09 的表达式依赖与 semantics_report.metrics 稳定。
5. unknown / ambiguous / parse failed 等诊断不会被吞掉。
```

## 2. Golden Case 目录结构

建议放在：

```text
backend/tests/golden_cases/
  c03_single_table_basic/
    input.sql
    metadata.json
    options.json
    expected.min.json
    README.md
  c04_join_alias_basic/
    input.sql
    metadata.json
    options.json
    expected.min.json
    README.md
  c08_query_result_final/
    input.sql
    metadata.json
    options.json
    expected.min.json
    README.md
  c09_metric_expression_basic/
    input.sql
    metadata.json
    options.json
    expected.min.json
    README.md
```

早期不要强制全量 JSON 完全相等，优先做“最小不变量断言”。

## 3. expected.min.json 格式

```json
{
  "case_id": "c09_metric_expression_basic",
  "expected_status": ["success", "partial"],
  "must_have_nodes": [
    {"id": "output_column:gmv", "type": "output_field"},
    {"id": "expression:gmv", "type": "expression"}
  ],
  "must_have_edges": [
    {"source": "column:dwd_order_di.order_amount", "target": "expression:gmv", "type": "expression_dependency"},
    {"source": "expression:gmv", "target": "output_column:gmv", "type": "expression_to_output"},
    {"source": "output_column:gmv", "target": "query_result:final", "type": "output_column_to_result"}
  ],
  "must_have_metrics": [
    {
      "name": "gmv",
      "depends_on_contains": ["order_amount"],
      "aggregate_functions_contains": ["SUM"]
    }
  ],
  "forbidden_diagnostics": ["PARSE_ERROR"],
  "invariants": ["no_dangling_edges"]
}
```

## 4. 测试策略

### 4.1 后端 pytest

核心测试文件：

```text
backend/tests/golden/test_c10_golden_cases.py
```

测试逻辑：

```text
1. 遍历 backend/tests/golden_cases/*。
2. 读取 input.sql / metadata.json / options.json / expected.min.json。
3. 调用现有 analyze service 或 TestClient POST /api/sql/analyze。
4. 按 expected.min.json 做子集断言。
5. 检查所有 edge.source / edge.target 都能在 nodes 中找到。
```

### 4.2 前端测试

建议保留轻量测试：

```text
frontend/src/components/__tests__/DetailPanel.test.tsx
frontend/src/__tests__/analyzeFlow.test.tsx
```

检查点：

```text
1. 传入 semantics_report.metrics 后，DetailPanel 能渲染表达式。
2. 点击 output 节点能找到对应 metric。
3. 点击 expression 节点能展示 node.data.expression。
4. 没有 semantics_report 时页面不崩溃。
```

## 5. C10 最小 Golden Case 清单

| case_id | 覆盖能力 | 必须断言 |
|---|---|---|
| c03_single_table_basic | 单表字段血缘 | column -> output -> Query Result |
| c04_join_alias_basic | Join 别名消歧 | u.id / o.user_id 来源不混乱 |
| c05_cte_structure_basic | CTE 结构依赖 | CTE 节点与依赖边存在 |
| c07_unknown_column_basic | unknown 诊断 | UNKNOWN_COLUMN 不被吞掉 |
| c08_query_result_final | 最终查询节点 | output_column_to_result 存在 |
| c09_metric_expression_basic | 表达式依赖 | expression_dependency / metric depends_on 完整 |

## 6. 验收命令

```bash
cd backend
pytest tests/golden/test_c10_golden_cases.py -q
pytest tests/integration -q

cd ../frontend
npm test -- --run
```

## 7. 禁止越界

```text
1. C10 不新增解析特性。
2. 不删除已有测试来“过关”。
3. 不用全量快照锁死仍在迭代的非核心字段。
4. 不忽略 diagnostics。
5. 不允许 graph edge 悬空。
```

## 8. C10 通过标准

```text
1. 所有 golden case 通过。
2. C09 表达式依赖 case 通过。
3. C08 query_result:final case 通过。
4. 后端 uvicorn 可启动。
5. 前端可以渲染 DetailPanel，不因 semantics_report 缺失崩溃。
```
