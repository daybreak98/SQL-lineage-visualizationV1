# C09｜表达式依赖与口径详情面板设计方案

## 1. 目标

C09 的目标是让系统从“字段从哪里来”进一步提升到“指标怎么算”。

在 C08 已经修复字段汇总链路后，字段级图已经具备：

```text
physical_column -> output_column -> Query Result
```

C09 在复杂输出字段中插入表达式节点：

```text
physical_column -> expression -> output_column -> Query Result
```

示例：

```sql
select
  sum(order_amount) as gmv,
  count(distinct order_no) as order_cnt,
  sum(order_amount) / count(distinct order_no) as adr
from dwd_order_di;
```

期望识别：

```text
gmv       = SUM(order_amount)，依赖 dwd_order_di.order_amount
order_cnt = COUNT(DISTINCT order_no)，依赖 dwd_order_di.order_no
adr       = SUM(order_amount) / COUNT(DISTINCT order_no)，依赖 order_amount + order_no
```

## 2. 后端新增能力

| 能力 | 说明 |
|---|---|
| expression node | 为聚合、函数、算术、case when 等表达式创建表达式节点 |
| expression_dependency edge | 源字段到表达式节点的依赖边 |
| expression_to_output edge | 表达式节点到输出字段节点的边 |
| semantics_report.metrics | 输出指标口径结构化说明 |
| aggregate function detection | 识别 sum/count/count distinct/avg/min/max |
| arithmetic operator detection | 识别 + - * / 等算术组合 |

## 3. API 响应新增字段

建议在现有 `AnalysisResult` 中增加：

```json
{
  "semantics_report": {
    "metrics": [
      {
        "name": "adr",
        "entity_id": "output_column:adr",
        "expression": "SUM(order_amount) / COUNT(DISTINCT order_no)",
        "depends_on": ["dwd_order_di.order_amount", "dwd_order_di.order_no"],
        "aggregate_functions": ["SUM", "COUNT_DISTINCT"],
        "operators": ["DIV"],
        "description": "由表达式 SUM(order_amount) / COUNT(DISTINCT order_no) 计算得到。",
        "evidence": {
          "source": "sqlglot_ast",
          "output_name": "adr"
        },
        "confidence_level": "high"
      }
    ]
  }
}
```

注意：`description` 只能是模板化确定性描述，不允许大模型生成。

## 5. 推荐算法

```text
for each select projection:
  1. 获取输出名 alias_or_name
  2. 获取真实表达式 expr
  3. 提取 expr 中的 Column 节点
  4. 调用现有 NameResolver，把 Column 解析成稳定字段名
  5. 提取聚合函数、算术操作符、函数名
  6. 判断是否需要 expression node
  7. 写入 semantics_report.metrics
  8. GraphBuilder 补 expression node / expression_dependency / expression_to_output
```

## 6. 是否创建 expression node 的规则

| 场景 | 是否创建 expression node | 说明 |
|---|---:|---|
| `select order_no from t` | 否 | 直接字段投影，沿用 C03/C08 |
| `select order_no as id from t` | 可不创建 | 简单 alias，直接 column_lineage 即可 |
| `select sum(amount) as gmv` | 是 | 聚合指标 |
| `select count(distinct order_no) as order_cnt` | 是 | 去重计数指标 |
| `select gmv / order_cnt as adr` | 是 | 算术派生指标 |
| `select case when ... end as tag` | 是 | 条件表达式 |
| `select get_json_object(data, '$.x') as x` | 是 | 函数派生字段 |

## 7. 前端 DetailPanel

点击 `output_field` 或 `expression` 节点时，DetailPanel 展示：

```text
字段名
表达式
依赖字段
聚合函数
算术操作符
诊断信息
字段注释
```

展示原则：

```text
1. 如果点击 expression 节点：直接展示 node.data.expression_detail。
2. 如果点击 output_field 节点：按 output name 或 entity_id 去 semantics_report.metrics 找对应 metric。
3. 解释文本仅作为辅助说明；真正血缘证据仍然来自 graph edges。
```

## 8. 验收标准

```text
1. sum(order_amount) as gmv → depends_on 包含 order_amount。
2. count(distinct order_no) as order_cnt → depends_on 包含 order_no。
3. sum(order_amount) / count(distinct order_no) as adr → depends_on 包含 order_amount 与 order_no。
4. expression_dependency edge.source / target 不悬空。
5. expression_to_output edge.source / target 不悬空。
6. DetailPanel 点击 output 或 expression 都能展示表达式和依赖字段。
7. C08 的 output_column -> query_result:final 不被破坏。
```

## 9. 禁止越界

```text
1. 不接入 AI / LLM。
2. 不做自然语言自由解释。
3. 不做完整语义层。
4. 不做 SQL 自动修复。
5. 不为所有未知复杂函数强行编造含义。
6. 不破坏 C08 的 Query Result 汇总链路。
```
