## 本轮目标

参考设计包：

`D:\Users\yingjie.hu\Documents\sql血缘解析可视化\SQL-lineage-visualizationV1\docs\sql_lineage_cleaning_design_pkg\sql_lineage_cleaning_design_pkg`

对项目现有复杂 SQL 处理链路做“最小增量优化”，重点不是新增一套 parser，而是补齐生产壳 SQL 的前置清洗与主查询选择能力，提升这类脚本的可解析率：

- 多语句脚本
- `set`
- `add jar`
- `use`
- `create temporary function`
- `insert overwrite ... select ...`

## 现状分析结论

当前仓库已经具备较完整的复杂 SQL guard：

- `preflight`
- `DirtySqlPreprocessor`
- `segmenter`
- 多轮 parse attempt
- segment parse fallback

现有问题不在 parser 本身，而在于：

- 生产脚本往往不是单条纯查询 SQL
- 真正需要送入 parser 的 analyzable query 没有被稳定选出来
- `insert overwrite table ... partition (...) select ...` 这种语句，目前更适合先抽出 `select` 部分再交给现有解析链路

## 已完成的开发内容

### 1. 新增轻量脚本清洗模块

已新增文件：

`backend/app/complex_sql_guard/script_cleaner.py`

当前这个模块是“最小实现”，目标是先完成主查询选择，不复制设计包整套实现。

已包含能力：

- 多语句按分号切分
- 识别并跳过：
  - `set`
  - `add jar`
  - `use`
  - `create temporary function`
  - 常见 `ddl`
- 识别普通 `query`
- 识别 `insert` 语句
- 对 `insert ... partition (...) select ...` 抽出真正需要分析的 `source query`
- 产出结构化选择结果：
  - `analysis_sql`
  - `start_offset`
  - `end_offset`
  - `selected_target`
  - `selected_kind`
  - `statement_count`
  - `skipped_count`

### 2. 已接入现有 complex_sql_guard 主链

已完成接入文件：

`backend/app/complex_sql_guard/analyzer.py`

当前接法：

1. 先对原始整段 SQL 做 `preflight`
2. 再执行 `statement_clean`
3. 选出真正需要分析的 query
4. 对选中的 query 执行：
   - `preprocess`
   - `segment`
   - `parse_attempts`

这样避免把整段生产脚本直接喂给 parser，同时保留原有复杂 SQL guard 主体设计。

### 3. 已处理 offset / location 回写

这一步已经完成，不再是待做项。

原因：

- 选中的 query 往往只是原始脚本中的一段
- 如果不做 offset rebase，后续 diagnostics / placeholder location / source mapping 会漂移

当前已做：

- `preprocess` 产出的 placeholder location 会回写到原始整段 SQL 坐标
- `preprocess` 产出的 diagnostics location 会回写到原始整段 SQL 坐标
- `segmenter` 继续基于原始 SQL 进行 location 映射

### 4. 已加入的诊断语义

这个模块已开始使用新的清洗阶段诊断语义，准备后续接入现有 `complex_sql_guard`：

- 多语句脚本检测
- 非分析语句跳过
- temp udf 注册提示
- insert 目标提取
- insert partition 检测
- 主分析语句选择

这些 code 已补入：

`backend/app/complex_sql_guard/diagnostics.py`
