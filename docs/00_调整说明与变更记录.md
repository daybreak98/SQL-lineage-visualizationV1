# 00｜调整说明与变更记录

> 本文用于说明：这次不是推翻原方案，而是把原方案从“Agent 快速重写型”改造成“Python 初学者可逐步实现型”。

---

## 1. 总体调整结论

### 1.1 主开发入口调整

(原：`01_Agent模块拆解文档/` 是当前最高优先级入口，按 M01-M28 模块拆解推进。)  
**新：`05_每轮开发文档/` 作为最高优先级入口，按 C00-C10 的纵向小闭环推进。**

原因：原 M01-M28 是横向模块拆分，适合经验开发者或 Agent 并行开发；Python 初学者更需要“先跑起来 → 前端能点 → 再加功能”的纵向反馈路径。

---

### 1.2 后端实现策略调整

(原：首轮目标倾向于快速恢复完整核心列血缘图，C00-C07 紧密连做。)  
**新：首轮只保证 FastAPI 可运行和前端可触发，真实血缘从 C02 开始逐步接入。**

原因：初学者如果一开始同时面对 FastAPI、Pydantic、SQLGlot、GraphViewModel、前端状态，会难以定位错误。拆小后，每轮只有一个主要错误来源。

---

### 1.3 前端对接方式调整

(原：前端文档重点描述 v1.4 最终交互效果，包括 Analyze → Search → Path → Detail → Locate SQL 的完整闭环。)  
**新：每轮只要求一个前端可见效果，例如 Health 绿灯、Analyze partial、画布出现首个节点、搜索结果出现输出字段、Locate SQL 高亮等。**

原因：前端最终体验不变，但开发节奏要从“最终形态倒推”改为“每轮都有可见反馈”。这样初学者可以通过页面结果判断后端是否真的生效。

---

### 1.4 测试策略调整

(原：强调 Contract Test + Golden Case + Snapshot Test 横向覆盖。)  
**新：每轮采用最小测试组合：1 个接口测试 + 1 个领域函数测试 + 1 个前端联调验收点；到 C10 再集中做 Golden Case 回归。**

原因：测试体系不能省，但初期不应一次性引入过多测试类型。先建立“改一点测一点”的习惯，再补全完整回归。

---

## 3. 保留不变的核心原则

以下原则不变：

1. 以后端为血缘结论来源，前端只展示和交互。
2. 不直接把 SQLGlot AST 暴露给前端。
3. CTE、子查询、输出字段不能伪装成物理表。
4. unknown / ambiguous / unsupported 必须结构化输出。
5. GraphViewModel 只包含图数据，不保存 selected / collapsed / viewport 等前端交互状态。
6. SQLite 元数据是补全能力，不是第一轮阻塞项。
