## 发现的问题

### P1：前端列元数据查询参数与后端不一致

现象：

```text
GET /api/metadata/columns?table=dwd_order_di
返回 total: 0
```

但后端实际接口参数是 `table_name`：

```text
GET /api/metadata/columns?table_name=dwd_order_di
返回 total: 3
```

影响：

```text
Metadata 弹窗或前端列元数据列表可能无法展示选中表的字段。
```

疑似位置：

```text
frontend/src/api/client.ts
backend/app/api/metadata_controller.py
```

建议后续修复：

```text
统一前后端参数名。推荐后端兼容 table 与 table_name，或前端改为 table_name。
```

### P2：tables_extracted 在多类 SQL 中为空

现象：

浏览器真实分析结果中，图模型已经包含表节点，但顶层字段：

```text
tables_extracted: []
```

在单表、Join、CTE 等场景均出现。

影响：

```text
如果前端或后续报告依赖 tables_extracted，会拿不到已识别表名。
```

建议后续修复：

```text
检查 parse_result.tables_extracted 或 assemble_result 的赋值链路，确保它与图模型中的 physical_table 节点一致。
```

### P3：浏览器长脚本容易超时

现象：

一次性执行完整 UI 多模块脚本时，browser-harness `Runtime.evaluate` 超时。

影响：

```text
不影响产品功能，但影响浏览器验收效率。
```

建议后续处理：

```text
继续使用 frontend/e2e/lineage_browser_smoke.py 这种短脚本模式。
复杂验收拆成多个 case，不要把所有 UI 操作塞进一次 Runtime.evaluate。
```
