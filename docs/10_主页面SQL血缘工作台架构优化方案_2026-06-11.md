# 主页面 SQL 血缘工作台架构优化方案

日期：2026-06-11  
范围：主页面 `Workbench`，不包含 SQL 方言转换次页面

## 1. 背景

当前主页面 SQL 血缘解析工作台已经具备完整闭环：

- SQL 编辑
- 后端分析请求
- 图数据归一化
- 多视图切换
- 节点/边选择
- 详情面板
- 搜索
- SourceLocation 联动
- 元数据导入弹窗

这说明主链路是通的，整体设计方向没有问题。

但随着功能继续增加，前端结构已经开始出现以下维护压力：

- `App.tsx` 承担的职责过多
- `WorkbenchState` 的状态写入点分散在多个组件中
- 图数据协议适配、视图裁剪、布局与搜索索引集中在 `graphPipeline.ts`
- `selectors.ts` 混入了运行时选择器、状态迁移描述、测试辅助和 mock 逻辑

如果继续在当前结构上直接叠功能，后续修改会越来越依赖人工记忆状态不变量，回归风险会持续增大。

## 2. 当前结构判断

### 2.1 合理的部分

- 后端统一输出 `graph_view_model`，前端再适配成内部图结构，这一层次是合理的
- 主页面和 SQL 方言转换页面已经做了页面级拆分，没有把两套任务流硬塞在一起
- 主页面图渲染采用“同一份 backend graph 派生不同可见图”的思路，这是正确方向
- SourceLocation、诊断、语义信息都已经是附加数据，而不是嵌死在节点结构里

### 2.2 需要优化的部分

#### A. `App.tsx` 过重

当前 `App.tsx` 同时负责：

- 页面入口切换
- Workbench 主状态持有
- 健康检查
- 元数据状态刷新
- SQL 修改后的脏状态迁移
- Analyze 成功/失败状态归并
- 搜索结果选择后的状态迁移
- 顶层组件拼装

这会导致：

- 任何工作台行为修改都要先读 `App.tsx`
- 状态迁移逻辑难以单测
- 后续多人并行修改时容易冲突

#### B. 状态迁移入口分散

`selectedEntity / selectedOutput / detailMode / renderMode / positions / drawerOpen` 等状态现在分别在：

- `App.tsx`
- `LineageCanvas.tsx`
- `CanvasToolbar.tsx`
- `DetailPanel.tsx`
- `Drawer.tsx`

中直接拼 patch 写入。

问题不是不能工作，而是：

- 很难保证每条交互路径都遵守同样的不变量
- 一个字段语义变化时，需要全局搜多处 patch
- `transitionRenderMode(...)` 已经不是唯一状态入口

#### C. 图处理模块职责过大

`graphPipeline.ts` 当前同时处理：

- backend graph 归一化
- 终点输出边补齐
- 搜索索引构造
- `table / subquery / column` 三类可见图裁剪
- 局部布局逻辑

这会让新视图、新布局或新搜索规则越来越难单独迭代。

#### D. 运行时逻辑与测试辅助逻辑混放

`selectors.ts` 中仍存在：

- 运行时选择器
- 状态迁移描述
- mock field graph 生成器
- 空实现占位函数

这会模糊“线上逻辑”和“测试遗留”的边界。

## 3. 优化原则

本轮及后续优化遵循以下原则：

- 不重写现有页面
- 不引入过度抽象
- 优先收敛状态边界，而不是先做视觉层改造
- 每一轮只做一个小闭环，并补进度文档
- 优先减少重复状态 patch，而不是追求一次性完美拆分

## 4. 推荐优化顺序

### 第一步：抽离主工作台状态模块

目标：

- 把 `App.tsx` 中的初始状态和核心状态迁移逻辑抽到独立模块
- 先统一“分析成功 / 分析失败 / SQL 修改 / 搜索选择”四类高频迁移

预期收益：

- `App.tsx` 变轻
- 状态迁移更容易复用和测试
- 为下一步 reducer/action 化打基础

### 第二步：收敛工作台 action

目标：

- 把跨组件常见 patch 写法逐步收敛成 action
- 组件只表达意图，不直接拼大对象 patch

可收敛 action 示例：

- `selectNode`
- `selectEdgeMapping`
- `clearSelection`
- `switchGraphViewMode`
- `resetViewport`
- `toggleDetailMode`
- `resetWorkspace`

### 第三步：拆分图处理模块

建议把 `graphPipeline.ts` 拆成：

- `graphAdapter.ts`
- `graphViews.ts`
- `graphSearch.ts`
- `graphLayout.ts`

目标：

- backend 协议适配、视图裁剪、搜索索引、布局彼此解耦

### 第四步：清理 selector 与测试夹杂结构

目标：

- 把 mock graph fixture 与运行时 selector 分离
- 删除空壳函数
- 明确运行时代码与测试辅助代码边界

## 5. 本轮准备实施的最小改造

本轮只做第一步的第一小点：

- 新增 Workbench 状态模块
- 抽离 `App.tsx` 中以下逻辑：
  - 初始状态
  - backend diagnostic 归一化
  - SQL 编辑后的脏状态迁移
  - analyze 成功态归并
  - analyze 失败态归并
  - 搜索结果选择后的状态迁移

本轮明确不做：

- 全量 reducer 改造
- 所有组件统一 action 化
- `graphPipeline.ts` 拆文件
- 复杂 UI 行为重构

## 6. 验收标准

完成本轮后，至少应满足：

- `App.tsx` 变薄，核心工作台状态迁移不再散落在页面主体内部
- 主页面现有分析、搜索、视图切换、详情联动行为不回归
- 有独立进度记录文档
- 后续提交 git 时，优化文档和进度文档一并进入 `docs`

## 7. 后续建议

等第一步稳定后，下一轮优先做：

1. Workbench action 收敛
2. `graphPipeline.ts` 拆分
3. `selectors.ts` 与 `mockLineage.ts` 清理

相比直接继续加功能，这三步会更明显地降低主页面后续复杂 SQL 能力扩展时的维护成本。
