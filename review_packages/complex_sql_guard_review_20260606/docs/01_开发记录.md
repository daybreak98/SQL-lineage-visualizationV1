# 复杂 SQL 解析容错能力开发记录

## 1. 背景与目标

当前项目已有简单 SQL 的表级关系、CTE 结构和部分列血缘能力，但面对生产环境中的复杂 Hive/Spark SQL 时，存在几个明显问题：

- 原始 SQL 中包含正则、JSONPath、模板变量、Freemarker 块、hint、注释时，`sqlglot` 很容易在入口阶段直接失败。
- 一旦完整 parse 失败，旧链路会把整条 SQL 判为失败，无法保留可恢复的结构信息。
- 复杂 SQL 缺少统一的预检、分段、降级、诊断框架，导致后续字段级血缘扩展缺少稳固基础。

本次改造的目标不是一次性完成完整字段级血缘，而是先在现有后端前面建立一层“复杂 SQL 容错处理层”，让系统具备以下能力：

- 永远保留 `original_sql`
- 生成 `normalized_sql` 和 `analysis_sql`
- 对 regex literal、JSONPath、模板变量、Freemarker、hint/comment 做可回溯 shielding
- 在完整 parse 失败时尝试 segment-level fallback
- 返回结构化 diagnostics、capabilities、confidence、segments、parse_attempts
- 尽量不破坏现有 `/api/sql/analyze` 与现有回归测试

## 2. 设计原则

本次开发严格遵守以下工程边界：

- 不直接重写现有 lineage engine，而是在现有解析服务前增加防御层。
- 不删除 `original_sql`，只新增 `normalized_sql` 和 `analysis_sql`。
- 不用粗暴全局 replace 破坏 SQL 结构，而是保留 placeholder 和 offset mapping。
- 简单坏 SQL 继续返回 `failed`，复杂但局部可恢复 SQL 才返回 `partial`。
- 对当前 resolver 明确不支持的结构，例如 `lateral view/explode`，不造假血缘，只返回诊断和降级结果。

## 3. 开发过程

### 3.1 梳理现有入口与风险点

首先定位了现有后端入口：

- `backend/app/api/analyze_controller.py`
- `backend/app/services/sql_parse_service.py`
- `backend/app/services/name_resolver.py`
- `backend/app/services/table_structure_service.py`
- `backend/app/services/cte_structure_service.py`

确认旧逻辑的主要问题是：

- `parse_sql()` 直接依赖 `sqlglot.parse_one`
- parse 失败时整体 `failed`
- 后续 graph / lineage 服务普遍假设必须拿到完整 tree

### 3.2 新增 `complex_sql_guard` 模块

在 `backend/app/complex_sql_guard/` 下新增了以下模块：

- `models.py`
- `diagnostics.py`
- `dialect.py`
- `normalizer.py`
- `preflight.py`
- `shields.py`
- `segmenter.py`
- `parser_adapter.py`
- `analyzer.py`
- `__init__.py`

这层模块独立于现有业务服务，职责清晰，后续可继续承接字段级血缘扩展。

### 3.3 预检阶段 `preflight`

在 `preflight.py` 中实现了 SQL 的轻量级预检，不尝试解析 AST，只做风险信号识别：

- 长度和行数
- 模板变量
- Freemarker 块
- regex/json 函数
- row-expanding function
- 括号平衡
- 引号平衡
- 隐形字符
- 复杂度评分

预检结果统一汇总为 `PreflightReport`，不会直接抛异常。

### 3.4 shield 机制

在 `shields.py` 中实现了 `DirtySqlPreprocessor`，核心动作是：

- 先做最小可逆规范化，生成 `normalized_sql`
- 在 `analysis_sql` 上屏蔽：
  - string literal
  - regex literal
  - JSONPath literal
  - `${...}` / `#{...}`
  - Freemarker block marker
  - comment / hint
- 为每个 placeholder 保留：
  - placeholder 名称
  - kind
  - raw_text
  - original source location

同时新增 `OffsetMapping`，为后续 source location 回溯保留基础。

### 3.5 SQL 分段 `segmenter`

在 `segmenter.py` 中实现了基于顶层 token、括号深度和 clause 边界的分段逻辑，支持识别：

- `cte_block`
- `cte_item`
- `main_select`
- `select_list`
- `from_join`
- `from_source`
- `join_block`
- `join_condition`
- `where`
- `group_by`
- `having`
- `order_by`
- `union_branch`
- `lateral_view`

这里没有走“按行切分”的错误方式，而是基于顶层结构切段。

### 3.6 parser adapter 与 analyzer 总编排

在 `parser_adapter.py` 中将 `sqlglot` 调用封装成统一的 `ParserAdapter`：

- 隔离 `sqlglot` 依赖
- 捕获 parse 异常
- 返回结构化 `ParseAttempt`
- 为 `sqlglot` 不存在的场景预留 `SQLGLOT_NOT_INSTALLED`

在 `analyzer.py` 中实现总入口 `analyze_complex_sql()`，执行顺序为：

1. preflight
2. preprocess / shields
3. segment
4. 依次尝试 parse:
   - `original_sql`
   - `normalized_sql`
   - `analysis_sql`
5. 如果完整 parse 失败，执行 segment-level fallback
6. 汇总 diagnostics / capabilities / confidence / unsupported_features

### 3.7 接入现有解析入口

对现有后端的接入重点落在：

- `backend/app/services/sql_parse_service.py`
- `backend/app/api/analyze_controller.py`
- `backend/app/adapters/sqlglot_adapter.py`
- `backend/app/models.py`
- `backend/app/domain/diagnostics_model.py`

实现方式是：

- `parse_sql()` 改为先调用 `analyze_complex_sql()`
- 若已得到完整 tree，则继续复用现有 graph/lineage 逻辑
- 若没有完整 tree，但 segment 可恢复，则返回 `partial`
- 保留旧模型中的 `status / diagnostics_report / output_fields / graph_view_model`
- 新增返回：
  - `analysis_sql`
  - `sql_text_bundle`
  - `preflight_report`
  - `segments`
  - `parse_attempts`
  - `capabilities`
  - `confidence`
  - 顶层 `diagnostics`

### 3.8 兼容性处理

为了不破坏现有回归，补了以下兼容策略：

- 简单坏 SQL，例如 `select from`，仍保持 `failed`
- `sql_parse` stage 仍放在 stage 列表首位
- 旧诊断码 `SQL_PARSE_ERROR` 保留
- 成功通过 `analysis_sql` 恢复解析的场景，不再误报整体失败

### 3.9 防止错误血缘

在 `name_resolver.py` 中新增了对 `exp.Lateral` 的防御判断：

- 检测到 `lateral view / explode` 时不继续做假定型列血缘
- 直接返回 `UNSUPPORTED_LATERAL_VIEW`

这样保证“不能精确支持时返回诊断”，而不是“猜出错误结果”。

### 3.10 测试隔离修复

在全量回归时，发现原有测试共享 SQLite 元数据库，导致 metadata 测试和 analyze 测试之间相互污染。

因此在 `backend/tests/conftest.py` 中补充了自动数据库清理：

- 每个测试前运行 migration
- 清空 `metadata_imports / table_metadata / column_metadata`

这一步不是复杂 SQL 逻辑本身，但它是让全量回归稳定通过的必要修复。

## 4. 本次实际交付

### 新增能力

- Complex SQL Guard 前置容错链路
- shield / segment / parse fallback / diagnostics
- API 扩展字段
- 复杂 SQL golden cases
- 后端专项测试与回归稳定性修复

### 关键结果

- 简单 SQL 解析能力保持可用
- 模板变量不会直接打崩解析入口
- regex / JSONPath 会被 shield 后继续分析
- 单个损坏的 CTE 不再导致整条 SQL 直接失败
- `lateral view/explode` 至少可识别、可诊断、可降级

## 6. 已知限制

- 当前仍不是完整字段级血缘引擎。
- `lateral view / explode` 只做到识别和降级，没有精确列血缘。
- 黑盒 UDF 当前只给 `BLACK_BOX_UDF` 诊断，没有表达式级依赖展开。
- `offset_mapping` 已建立，但尚未被下游 graph/source location 全面消费。

## 7. 后续建议

- 下一步优先把 `analysis_sql + placeholders + offset_mapping` 接到字段级 resolver。
- 为 `explode / posexplode / json_tuple / from_json / black-box udf` 建专门的 lineage 规则。
- 将 golden cases 接入 CI，避免后续升级 `sqlglot` 时回归失控。
